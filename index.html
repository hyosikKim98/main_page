Introduction

기능을 만드는 것보다, 이후에도 흔들리지 않게 만드는 일에 더 관심이 있습니다.

고등학교 시절 간단한 공 튀기기 게임을 구현하면서, 복잡한 움직임도 규칙으로 분해하면 설계로 이어진다는 감각을 처음 얻었습니다. 이후 프로그래밍을 전공하고 IoT를 복수전공하며 현실의 문제를 안정적으로 해결하는 개발자를 목표로 학습해 왔습니다.

프로젝트를 거치며 기능 구현만으로는 충분하지 않다는 점을 반복해서 확인했습니다. 요청이 몰리는 구간, 실패가 섞이는 구간, 사용 흐름이 반복되는 구간을 먼저 정의하고 구조를 정리해야 이후에도 유지되는 시스템을 만들 수 있다고 생각합니다.

Principles

제가 작업을 정리하는 기준

01

문제를 흐름 단위로 봅니다

병목이 생기는 지점과 정합성이 깨지는 지점을 먼저 나눠 보고, 기능보다 요청 흐름 전체를 기준으로 우선순위를 정합니다.

02

구조로 불안정을 줄입니다

무거운 작업은 사용자 요청에서 분리하고, 책임을 역할별로 나눠 장애가 생겨도 원인을 빠르게 추적할 수 있는 구조를 선호합니다.

03

선택은 검증으로 마무리합니다

구조적으로 그럴듯한 해법을 그대로 유지하지 않고, 테스트와 지표를 통해 현재 환경에서 실제로 더 나은 선택인지 확인합니다.

Selected Work

혼잡 구간과 실패 구간을 제어한 프로젝트

Ticketing Server

대규모 트래픽에서도 예매 흐름이 끊기지 않도록 제어한 티켓팅 서버

  • Redis ZSET 대기열과 최대 20명 활성 슬롯으로 결제 구간 진입량을 통제했습니다.
  • Kafka 비동기 처리로 요청 접수와 재고 차감을 분리해 병목이 전체 API로 번지지 않게 했습니다.
  • JWT, 관리자 전용 발급 API, idempotency key와 TTL 가드로 권한과 중복 결제를 방어했습니다.
  • Prometheus, Grafana, JMeter로 대기열 길이와 슬롯 반환이 의도대로 유지되는지 확인했습니다.
Spring Boot Redis Kafka Prometheus

Commerce API

동시 주문 상황에서도 재고 무결성을 지킨 쇼핑몰 백엔드

  • 상품별 Redis 락과 DB 조건부 차감으로 애플리케이션과 데이터베이스 양쪽에서 정합성을 방어했습니다.
  • 락 획득 실패는 429 Too Many Requests로 구분해 재시도 가능한 실패와 재고 부족을 분리했습니다.
  • 동시 주문 테스트에서 재고 100개, 요청 140건 상황에서도 oversell 없이 최종 재고 0개를 유지했습니다.
  • 이후 Redis 락 방식과 DB only 방식을 비교 측정해, 단일 인스턴스 환경에서는 더 단순한 경로가 적합함을 검증했습니다.
Spring Boot Redis Lock MySQL Grafana

ESG Prediction Service

무거운 연산을 분리해 응답 흐름을 안정화한 예측 서비스

  • 사용자 요청, 외부 API 및 크롤링, 예측 연산 서버를 분리해 책임을 역할별로 재설계했습니다.
  • Redis에서 최신 예측 결과를 먼저 확인하고, 갱신이 필요할 때만 Kafka 큐와 Celery 워커로 백그라운드 작업을 넘겼습니다.
  • 크롤링과 예측 연산이 사용자 요청과 동기적으로 묶여 생기던 지연을 제거해 타임아웃을 줄였습니다.
  • 요청이 몰려도 응답 흐름이 흔들리지 않도록 구조 자체를 바꾸는 데 집중했습니다.
Redis Kafka Celery Python

Strengths

프로젝트 밖에서도 유지되는 강점

정확성과 신뢰 중심의 태도

확인되지 않은 정보를 바로 전달하기보다 사실관계를 먼저 점검하고 정리하는 편입니다. 최근에는 확정된 사실을 먼저 공유하고, 추가 확인이 필요한 부분은 별도로 구분해 초기 속도까지 보완하고 있습니다.

변경 이력까지 고려한 데이터 구조화

월별 배출량 데이터를 JSON 구조로 재설계하고, 변경 시점과 수정 사용자를 함께 저장해 cell 단위 이력을 추적할 수 있게 했습니다. 원본 반영과 이력 기록은 트랜잭션으로 묶어 정합성을 확보했습니다.

사용자 동선을 먼저 줄이는 개선

기업 검색 흐름을 분석해 메인 → 검색 → 기업 선택의 3단계 동선을 메인에서 바로 선택하는 1단계로 줄였습니다. 기능 추가보다 반복 행동을 줄이는 편이 사용성 개선에 더 직접적이라고 판단했습니다.
