쇼핑몰 API 서버

1인 개발 · 2026.01 ~ 2026.02 · 백엔드 설계 및 구현

GitHub 저장소
개요
주문 정합성과 조회 성능을 함께 개선한 쇼핑몰 서버 프로젝트
주요 기능
  • 주문 동시성 제어를 통한 안정적인 재고 처리
  • 상품 목록 캐시 적용을 통한 빠른 조회 응답
  • 성능 테스트 및 지표 분석을 통한 기술 스택 비교 검증
기술
Spring Boot / Redis / PostgreSQL
문제

동시 주문 환경에서 재고 정합성 보장

문제 상황
동시 주문 시 동일 재고가 중복 차감되어 실제 재고와 시스템 재고 간 불일치 가능
접근 방식
상품 단위 락으로 동시성을 제어하고, 재고 중복 차감을 방지, 실패 원인을 락 획득 실패와 재고 부족으로 구분해 재시도 흐름까지 안정화
처리 흐름
주문 요청 → Redis Lock 획득 → 재고 차감 → 주문 생성 → Lock 해제 → 주문 성공 반환
락 실패 반환 재고 부족 반환
검증 결과
동시성 테스트에서도 음수 재고 없이 최종 재고 정합성을 유지, 실패 흐름 역시 락 실패와 재고 부족으로 명확히 분리
배운점/교훈
동시성 제어는 단순히 요청을 막는 문제가 아니라, 실패를 예측 가능하게 다루도록 설계해야하는 문제라는 점을 배웠습니다.
성능 비교

성능 비교를 통한 동시성 제어 방식 선택

검증 방식
동일 상품에 주문이 집중되는 경쟁 시나리오를 구성한 뒤, JMeter와 Grafana로 재고 차감 전략별 처리 성능을 비교 검증했습니다.
  1. Redis 락 + DB 조건부 차감
  2. DB 조건부 차감만 적용
검증 결과

현재 단일 인스턴스 구조에서는 DB 조건부 차감만 적용한 방식이 더 효율적이었습니다. [Grafana 관측 기준] 최대 처리량 : 약 1.5k req/s -> 4.3k req/s (약 2.9배) [Grafana 관측 기준] 응답 지연 : p95 200~450ms -> 150~220ms, p99 370~820ms -> 220~315ms

배운점/교훈
동시성 제어 방식은 복잡한 기술을 우선 적용하기보다, 현재 구조에서 가장 단순하면서도 성능과 정합성을 함께 만족하는 방식을 검증 후 선택해야 한다는 점을 배웠습니다.
문제

캐싱을 통한 반복 조회 성능 최적화

문제 상황
반복되는 목록 조회가 모두 DB로 향해 읽기 부하와 응답 지연이 발생 가능
접근 방식
검색 조건 기반 Redis 캐시와 TTL, 변경 시 캐시 무효화, Redis 장애 시 DB fallback을 적용
처리 흐름
상품 목록 조회 요청 → Redis 캐시 확인 → 캐시 hit 시 반환
캐시 miss → DB 조회 → 캐시 저장 → 조회 결과 반환
검증 결과

워밍 업 구간 제외

캐싱 - p95 약 18~21ms, p99 약 22~26ms 수준

비캐싱 - p95 약 22~27ms, p99 약 25~28ms 수준

배운점/교훈
캐싱은 응답 속도를 개선 기능이 뿐만 아니라, 어떤 조건을 키로 삼고 언제 무효화할지까지 포함한 데이터 일관성 설계라는 점을 배웠습니다.

GitHub 저장소
개요
대기열과 비동기 결제로 혼잡 상황의 예매 흐름을 제어한 서버 프로젝트
주요 기능
  • 대기열 기반 동시 접속 제어를 통한 안정적인 예매 처리
  • 비동기 결제 요청 처리와 트랜잭션 기반 재고 차감으로 정합성 확보
  • 중복 요청 방지 및 운영 모니터링을 통한 예매 시스템 안정성 강화
기술
Spring Boot / Kafka / Redis / PostgreSQL
문제

대규모 동시 접속 티켓팅 대응

문제 상황
예매 오픈 시 다수 사용자가 동시에 결제 구간에 몰리며 서버 병목, 중복 요청, 재고 정합성의 문제 발생 가능
접근 방식
Redis 대기열로 사용자 순번을 관리하고 입장 토큰으로 결제 가능 인원을 제한 Lua 스크립트로 토큰 발급을 원자적으로 처리 Kafka 비동기
처리 흐름
사용자 → 대기열 등록 → 입장 토큰 발급 (최대 20명) → 토큰 유효 확인 → 결제 이벤트 전송 → 결제 처리 / 재고 차감
만료 슬롯 정리 슬롯 반환
검증 결과
JMeter 200명 혼합 시나리오 기준

혼잡 구간에서 대기열은 일시적으로 약 140명까지 증가, 활성 슬롯은 20명 수준으로 안정적으로 유지 이후 대기열이 지속적으로 감소해 종료 시점에는 거의 해소되며, 동시 진입 제어와 처리 안정성을 확인

배운점/교훈
캐싱은 응답 속도를 개선 기능이 뿐만 아니라, 어떤 조건을 키로 삼고 언제 무효화할지까지 포함한 데이터 일관성 설계라는 점을 배웠습니다.
관측

대기열 기반 티켓팅 제어 : 대시보드

시나리오 : 대기열의 70% 결제(슬롯 반환), 30% 토큰 만료
